MODIFICATIONS APPLIED
================================================================================

Based on automation testing best practices, the following modifications were made:

EXCLUDED FROM AUTOMATION:
-------------------------
✗ Negative test cases (invalid inputs, duplicates, empty fields)
✗ Unit-level field validations (email format, password requirements)
✗ Security injection tests (SQL injection, XSS - detailed testing)
✗ Browser-specific validation tests (password masking)
✗ Session timeout testing (better for manual testing)

INCLUDED IN AUTOMATION SUITE:
-----------------------------
✓ Critical path (happy path) scenarios
✓ Complete CRUD operations (Create, Read, Update, Delete)
✓ End-to-end workflows
✓ Non-functional tests (NEW - added per requirements)
✓ Session management (logout flow)
✓ Data integrity validations

================================================================================
RATIONALE FOR CHANGES
================================================================================

WHY EXCLUDE NEGATIVE TESTS FROM AUTOMATION?
--------------------------------------------
1. Better suited for manual exploratory testing
2. Require human judgment for edge cases
3. Often change with UI/validation updates
4. Lower ROI for automation maintenance
5. Development team catches most via unit tests

WHY FOCUS ON CRITICAL PATH?
----------------------------
1. High business impact if broken
2. Stable workflows less prone to change
3. Represents real user journeys
4. Quick regression smoke tests
5. Essential for CI/CD pipeline

WHY ADD NON-FUNCTIONAL TESTS?
------------------------------
1. Ensures consistent user experience
2. Catches deployment/configuration issues
3. Validates frontend integrity
4. Easy to automate and maintain
5. Critical for production readiness
